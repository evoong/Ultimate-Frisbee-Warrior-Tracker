# Strategy board arrows: design

## Summary

Add the ability to draw arrows on the Strategy board to diagram player
movement and cuts. Arrows are freeform annotations on the field (not tied to
any player), persisted per play, and can be bent into a single curve so they
read as cutting plays. Arrows are drawn by toggling a Draw mode (works on
desktop and touch) or, on desktop, by holding the A key while dragging on empty
field space.

## Goals

- Draw an arrow by dragging on empty field space (no player in hand).
- Bend an arrow into one smooth curve for cuts, via a draggable midpoint handle.
- Make the feature discoverable (a toggle button with a tooltip, plus updated
  helper text).
- Persist arrows per play with the same public-read / team-write security model
  as the rest of the board.
- Work in both the desktop landscape frame and the mobile portrait frame.

## Non-goals (deliberately out of scope)

- Per-arrow colour picker: a single high-contrast colour for all arrows.
- Dashed vs solid or other line styles.
- Text labels on arrows.
- Undo/redo beyond deleting an arrow.
- Realtime sync between clients (last write wins, matching the existing board).

Each of these is easy to add later without reworking the data model.

## Data model

New migration `supabase-migrations/007_strategy_arrows.sql`. New migrations
start at 007 per CLAUDE.md.

Table `public.strategy_arrows`:

- `id bigint generated by default as identity primary key`
- `play_id bigint not null references public.strategy_plays(id) on delete cascade`
- `x1, y1 real not null check (>= 0 and <= 1)` : start point
- `x2, y2 real not null check (>= 0 and <= 1)` : end point
- `cx, cy real not null check (>= 0 and <= 1)` : quadratic Bezier control point
- `created_at timestamptz not null default now()`

Coordinates are fractions in [0, 1] of the canonical LANDSCAPE field, the same
frame as `strategy_positions` (x along the 100m length, y across the 37m
width). The frontend maps canonical coordinates to the portrait rendering on
mobile, so stored arrows rotate correctly with the board. Never store rendered
offsets.

A straight arrow stores `cx,cy` at the midpoint of the two endpoints. Bending
moves the control point. Storing the control point directly (rather than the
on-curve handle position) lets rendering feed the values straight into an SVG
quadratic path command.

RLS: same public-read / team-write shape as every other table (copy the
per-table template from migration 004 / 006). Authenticated users may read;
insert, update, and delete require `public.is_allowed()`.

## Hooks

Add to `frontend/hooks/backend/strategy.ts`, mirroring the existing position
hooks:

- `type StrategyArrow = { id: number; x1: number; y1: number; x2: number; y2: number; cx: number; cy: number }`
- `useGetStrategyArrows()` : select arrows for a play id, ordered by id.
- `useCreateStrategyArrow()` : insert one arrow, return the created row
  (including its server id) so the optimistic temp id can be reconciled.
- `useUpdateStrategyArrow()` : update one arrow by id (all six coordinates).
- `useDeleteStrategyArrow()` : delete one arrow by id. Returns true on success
  so the caller can revert on failure, matching `useDeleteStrategyPosition`.

## Page wiring (`frontend/pages/Strategy.tsx`)

- Hold arrows in local state as an array (`StrategyArrow[]`), loaded alongside
  positions whenever the selected play changes (clear first to avoid the
  previous play's arrows flashing).
- Pass `arrows`, `onCreateArrow`, `onUpdateArrow`, `onDeleteArrow` to
  `StrategyBoard`.
- Optimistic saves, matching the existing position pattern:
  - Create: append a local arrow with a temporary negative id, call the insert,
    then replace the temp id with the server id on success; on failure remove
    the temp arrow (or refetch).
  - Update: apply locally on drag release, call update, refetch arrows to revert
    on error.
  - Delete: remove locally, call delete, refetch to revert on error.
- Surface arrow save errors in the existing `saveError` line.

## Rendering (`frontend/components/strategy/StrategyBoard.tsx`)

An SVG overlay fills the field container, layered beneath the player avatars.

- `viewBox="0 0 100 37"` in landscape, `"0 0 37 100"` in portrait, so one unit
  equals one metre and stroke width and arrowheads scale uniformly with no
  distortion. Reuse the existing canonical -> rendered fraction mapping
  (`toRendered`) and multiply by the viewBox dimensions to place points.
- Each arrow renders as a `path` with `d="M x1 y1 Q cx cy x2 y2"` and an
  `orient="auto"` arrowhead `marker` at the end.
- One bold, high-contrast colour for all arrows, chosen to read on the emerald
  field in both light and dark themes.
- `pointer-events: none` on the `<svg>` element; `pointer-events` enabled on the
  individual arrow paths (with a wider transparent hit path if the visible
  stroke is thin) so arrows are clickable without blocking player drags
  elsewhere on the field.

## Interaction

### Modes

- A `Move | Draw arrow` toggle sits above the board, rendered only when
  `allowed`. Mode state lives inside `StrategyBoard` (co-located with the
  pointer and key handlers).
- In Draw mode, a pointer-down on empty field space starts an arrow; drag sets
  the end point; release creates a straight arrow (control point at the
  endpoints' midpoint) and auto-selects it.
- On desktop, holding the A key transiently arms the same draw behaviour without
  flipping the toggle. Key handling:
  - `keydown`/`keyup` for `a`/`A` on `window`, ignoring auto-repeat.
  - Ignore when the event target is an input, textarea, or contenteditable, or
    when a dialog is open, so typing a play name never arms drawing.
  - The A key is a desktop convenience only; the toggle is the primary,
    cross-device trigger.

### Player dragging vs arrow drawing

- Player avatars keep their own pointer handlers and sit above the SVG overlay,
  so Move mode is unchanged.
- Arrow drawing is driven by a pointer handler on the field container (or a
  dedicated overlay) that is active only when Draw mode is on or A is held.

### Selecting and editing

- Tapping an arrow selects it (`selectedArrowId` in board state). Tapping empty
  space (outside Draw mode) or pressing Escape deselects.
- A selected arrow shows three drag handles: start, end, and an on-curve
  midpoint handle.
  - Dragging start or end moves that endpoint.
  - Dragging the midpoint handle bends the arrow. The control point is solved so
    the curve passes through the handle: for a quadratic Bezier the on-curve
    midpoint is `0.25*P0 + 0.5*C + 0.25*P2`, so `C = 2*H - 0.5*(P0 + P2)` where
    `H` is the handle position. Keeping the handle on the curve makes bending
    feel direct.
  - Handles use `setPointerCapture` like the existing avatar drag, need
    `touch-none`, and clamp coordinates to [0, 1].
- A small circular delete button (an HTML button positioned by percentage near
  the arrow) removes the selected arrow.

### Saving

- Optimistic, matching positions: local state updates immediately; the write
  happens on create and on each edit's drag release; a failed write refetches
  arrows to revert. No realtime; last write wins.

## Discoverability

The explicit "make sure the user knows this is possible" requirement:

- Tooltip on the Draw arrow toggle: "Draw arrows for cuts and movement. Toggle
  on and drag on the field, or hold A and drag."
- Updated helper text under the board, extending the existing sentence:
  "Drag players from the bench onto the field. Drag a player off the field to
  bench them. Toggle Draw arrow (or hold A) and drag on the field to add cutting
  arrows."

## Testing

- Typecheck: `cd frontend && npx tsc --noEmit` (diff against touched files;
  pre-existing errors exist elsewhere).
- Manual verification on desktop (landscape) and a mobile viewport (portrait):
  - Draw a straight arrow via the toggle and via hold-A.
  - Bend it with the midpoint handle; confirm the curve persists across a
    reload and across play switches.
  - Move endpoints; delete an arrow; confirm arrows do not block player drags.
  - Confirm arrows rotate correctly between landscape and portrait.
  - Confirm a non-allowed (read-only) user sees arrows but no toggle or handles.
